01 · Background

왜 만들었나

계기는 단순했다. 아들의 기말고사 성적표를 받아보고 나서, "이대로는 안 되겠다"는 생각이 들었다. 국어 96점, 영어 80점인데 수학은 31점 — 과목별 편차가 너무 컸다. 그런데 세상에 있는 학습 앱들은 대부분 "매일 5과목 다 하기" 식의 획일적인 체크리스트였다.

그래서 방향을 세 가지로 잡았다.

  • 과목별로 다른 학습량 — 취약한 수학은 매일, 잘하는 과목은 유지 위주로
  • 체크리스트가 아니라 동기부여 장치 — 레벨업, 나무가 자라는 성장 은유, 배지, 그리고 진짜 용돈으로 이어지는 보상
  • 부모와 아이가 같은 데이터를 실시간으로 공유 — 억지로 검사하지 않아도 자연스럽게 확인할 수 있게

02 · Origin

시작은 이렇게 — 세 번의 시도

1차 · Codex

가장 먼저 시도한 도구. 하지만 만들어달라고 요청하고 문제를 짚으면, 그때마다 "사실은 다른 방법으로 해야 한다"며 계속 말을 바꿔서 진도가 나가지 않았다.

2차 · Figma Make

화면 하나씩은 그럭저럭 만들어졌지만, 여러 화면이 맞물려 동작하는 "진짜 앱"을 만드는 데는 한계가 있었다. Codex가 안내한 방식대로 요청해봐도, 계속 다른 방향으로 다시 안내하기를 반복하다 결국 더 이상 진행이 불가능해졌다.

3차 · Claude Code

두 번의 실패 끝에 정착한 도구. 대화로 기능을 요청하면 실제로 동작하는 화면까지 만들어졌고, 버그를 신고하면 원인을 찾아 고쳐가며 실제 배포까지 이어졌다.

04 · Lessons

바이브 코딩하면서 겪은 문제들

함정

그럴듯해 보이지만 가짜였던 기능

사진 인증 화면이 실제 카메라 없이 UI만 흉내 낸 채로 한동안 방치돼 있었다. 화면은 완벽해 보였지만, 직접 써보기 전엔 아무도 몰랐다.

함정

"됐다"는데 실제로는 안 되는 상태

미션을 체크해도 화면에 반영이 안 되는 버그가 있었는데, 코드 로직 문제가 아니라 "오늘 날짜가 일정 시작일 하루 전"이라는 아주 사소한 경계 조건이 원인이었다.

함정

예전 문구가 곳곳에 남음

기능을 바꿔도 예전 텍스트("42일 프로젝트" 등)가 다른 화면에 그대로 남아있어서, 새 기능을 추가할 때마다 관련 없어 보이는 화면까지 다시 훑어봐야 했다.

함정

초기화 버튼이 절반만 동작

"데이터 초기화"를 눌러도 일부 항목만 지워지고 나머지는 남아있는 식의 빠짐이 종종 있었다.

보안

아이가 부모 모드로 바로 바꿀 수 있었던 구멍

처음 설계에서는 설정에서 "부모 폰"을 그냥 눌러 바꿀 수 있었다. 용돈 지급 버튼이 걸려있는 기능이라 뒤늦게 PIN 인증을 추가해서 막았다.

대화로 요청하면 빠르게 만들어준다는 게 바이브 코딩의 강점이지만, 실제로 버튼을 눌러보고 화면으로 확인하지 않으면 "말로는 됐다는데 실제로는 안 되는" 부분을 놓치기 쉽다. 매번 직접 테스트하는 과정이 결국 가장 중요한 단계였다.

GitHub — 코드가 사는 곳

버전 관리 · 소스 코드 저장소

코드가 바뀔 때마다 "커밋"이라는 기록이 쌓이는 곳. Vercel이 배포할 때 이 저장소의 코드를 그대로 가져다 쓴다. GitHub 자체는 화면에 아무것도 안 보여준다 — 코드 창고일 뿐.

우리가 한 설정

  • jhkl0514/project-woohyun 이름으로 Public 저장소 생성 (github.com/new)
  • 로컬 프로젝트를 git remote add origin으로 연결 후 git push

다음에 새로 시작할 때

  1. github.com/new 에서 저장소 이름 정하고 Public(또는 Private) 선택 → Create
  2. 터미널에서 프로젝트 폴더로 이동 후 git init (이미 되어있으면 생략)
  3. git remote add origin https://github.com/계정명/저장소명.git
  4. git add . && git commit -m "first commit" && git push -u origin main

확인하는 곳 → github.com/본인계정/저장소명 에서 전체 코드와 커밋 기록 확인

Vercel — 실제로 서비스되는 곳

빌드 · 배포 · 호스팅

GitHub의 코드를 실제 웹사이트로 만들어 인터넷에 올려주는 서비스. 우리 앱 주소(project-woohyun.vercel.app)가 여기서 나온다. Supabase에 접속하기 위한 비밀 열쇠(환경변수)도 여기 등록해뒀다.

우리가 한 설정

  • 기존 Vercel 계정에 CLI로 프로젝트 연결 (vercel link)
  • 환경변수 2개 등록 — VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY (Supabase 접속 정보)
  • 개인 접근 토큰(Personal Access Token) 발급 후 CLI에 연결
  • vercel --prod 명령으로 실제 배포

개인 접근 토큰 — 왜 필요한가

Claude Code는 브라우저 로그인 창을 직접 띄울 수 없다. 그래서 vercel login 대신, 터미널(CLI)이 "이 사람이 맞다"고 증명할 수 있는 열쇠(토큰)를 미리 발급받아 매 명령마다 같이 제출하는 방식을 쓴다. 토큰만 있으면 브라우저 로그인 없이도 배포·환경변수 등록 같은 작업을 CLI로 대신 할 수 있다.

  1. vercel.com/account/settings/tokens 접속
  2. Create Token — 이름 정하고(예: claude-code), 만료 기간·범위(Scope, 어느 계정/팀에 쓸지) 선택 후 생성
  3. 생성된 토큰 문자열을 그 자리에서 바로 복사 (다시 못 봄 — 페이지를 벗어나면 재확인 불가)
  4. 이후 모든 vercel 명령 뒤에 --token 복사한값을 붙여서 실행
주의 — 이 토큰은 비밀번호와 같다. 이 토큰만 있으면 그 Vercel 계정으로 배포·삭제·환경변수 열람이 전부 가능하다. 다른 사람에게 공유하지 말고, 혹시 유출됐다면 같은 페이지에서 즉시 Revoke(폐기)할 수 있다.

다음에 새로 시작할 때

  1. vercel.com 가입/로그인
  2. 위 방식대로 토큰 발급 (또는 터미널에서 직접 vercel login)
  3. 프로젝트 폴더에서 vercel link (처음 한 번, 계정·프로젝트 이름 연결)
  4. Supabase 키가 있다면 vercel env add 변수이름 production으로 등록
  5. vercel --prod 로 배포 — 코드가 바뀔 때마다 이 명령을 다시 실행
주의 — 지금 이 프로젝트는 GitHub과 Vercel의 자동 배포 연결이 안 되어 있다. 즉 git push만 해서는 사이트에 반영되지 않고, vercel --prod를 직접 실행해야 실제로 배포된다.

확인하는 곳 → vercel.com 대시보드에서 배포 기록, 환경변수, 도메인 확인

Supabase — 데이터가 저장되는 곳

클라우드 데이터베이스

경험치·학습기록·사진·용돈처럼 가족이 공유해야 하는 데이터가 저장되는 곳. 로그인 없이 "가족 코드"라는 고유 번호 하나만으로 여러 기기가 같은 데이터를 보게 해주는 핵심 부품이다.

우리가 한 설정

  • family_data 테이블 하나 생성 — 가족 코드(id), 전체 데이터 묶음(payload), 마지막 수정 시각
  • 이 테이블에 행 단위 보안(RLS)을 켜고, 아무에게도 직접 접근 권한을 주지 않음
  • 대신 "정해진 통로" 역할을 하는 함수 2개만 외부에 공개 — get_family_data(읽기), upsert_family_data(쓰기)
  • 이렇게 한 이유: 앱에 박혀있는 공개 열쇠(anon key)를 누가 알아내도, 정확한 가족 코드를 모르면 어떤 가족의 데이터도 못 봄 — 전체 목록을 훑어보는 것 자체가 원천 차단됨

실제로 실행한 스크립트

create extension if not exists pgcrypto;

create table if not exists family_data (
  id uuid primary key default gen_random_uuid(),
  payload jsonb not null default '{}'::jsonb,
  updated_at timestamptz not null default now()
);

alter table family_data enable row level security;
-- 테이블에 대한 직접 select/insert/update 정책은 만들지 않음 (기본적으로 전부 차단)

create or replace function get_family_data(fam_id uuid)
returns jsonb language sql security definer set search_path = public
as $$ select payload from family_data where id = fam_id; $$;

create or replace function upsert_family_data(fam_id uuid, new_payload jsonb)
returns jsonb language sql security definer set search_path = public
as $$
  insert into family_data (id, payload, updated_at)
  values (fam_id, new_payload, now())
  on conflict (id) do update set payload = excluded.payload, updated_at = now()
  returning payload;
$$;

revoke all on function get_family_data(uuid) from public;
revoke all on function upsert_family_data(uuid, jsonb) from public;
grant execute on function get_family_data(uuid) to anon;
grant execute on function upsert_family_data(uuid, jsonb) to anon;

1단계 — 프로젝트 생성 (New Project)

supabase.com에 가입하면 Organization(계정 단위) 하나가 자동으로 생기고, 그 안에 프로젝트를 만든다. 우리 프로젝트를 만들 때 실제로 열었던 화면:

  1. supabase.com/dashboard/new/[조직ID] — "New Project" 생성 화면 (URL의 조직 ID는 계정마다 다르므로, 다음번엔 그냥 대시보드에서 New Project 버튼을 누르면 됨)
  2. Project name 정하기 (예: jhkl0514's Project)
  3. Database Password 설정 — 이건 앱이 쓰는 열쇠와는 별개로, DB에 직접 접속할 때 쓰는 관리자 비밀번호다. 꼭 어딘가에 저장해둘 것 (나중에 다시 확인 불가)
  4. Region은 한국에서 가장 가까운 지역(Northeast Asia, Seoul/Tokyo 등) 선택 — 속도에 영향
  5. Plan은 Free로 충분 — 지금 이 앱 정도 사용량이면 무료 한도 안에서 넉넉함

2단계 — 프로젝트 설정 확인 (Settings → General)

프로젝트를 만들고 나면 기본 정보를 확인하는 곳:

  1. Settings → General 접속
  2. 여기서 Project name(나중에 바꿀 수 있음), Reference ID(프로젝트마다 고유한 식별자 — 우리 프로젝트는 mqxniwjyehvmecnndysy, 앱 접속 주소에도 그대로 들어감), Region을 확인할 수 있다
  3. 당장 뭘 바꿀 필요는 없음 — "지금 내가 맞는 프로젝트에 들어와 있는지" 확인하는 용도로 주로 씀

3단계 — 접속 열쇠 복사 (API Keys)

우리 앱(브라우저에서 돌아가는 코드)이 이 데이터베이스에 접속하려면 "이 프로젝트의 주소"와 "공개해도 되는 열쇠"가 필요하다. 그걸 가져오는 곳:

  1. Settings → API Keys → Legacy API key 접속 (Supabase가 최근 새 키 체계를 도입해서 "Legacy"라고 표시되는데, 우리 앱은 이 기존 방식의 키를 쓴다)
  2. Project URL 복사 → VITE_SUPABASE_URL로 사용
  3. anon / public 키 복사 → VITE_SUPABASE_ANON_KEY로 사용
왜 anon 키는 복사해서 앱 코드에 넣어도 되는가 — 이 키는 원래 "누구나 봐도 되는" 공개 열쇠로 설계되어 있다. 대신 이 열쇠로 할 수 있는 일을 최소한으로 제한해뒀다 (테이블 직접 접근 금지, 정해진 함수 2개만 허용, 그마저도 정확한 가족 코드가 있어야 동작). 그래서 이 키가 웹사이트 코드 안에 그대로 보여도 — 개발자 도구로 누가 열어봐도 — 실제로 데이터를 빼갈 방법이 없다. (반대로 service_role 키는 모든 걸 다 할 수 있는 진짜 비밀 열쇠라 앱 코드에 절대 넣으면 안 된다 — 우리는 이 키를 쓰지 않았다.)

4단계 — 테이블·함수 만들기 (SQL Editor)

위 "실제로 실행한 스크립트"를 실행한 곳. 이 화면은 데이터베이스에 직접 명령을 내리는 곳이라, 여기서 실행해야 family_data 테이블과 데이터를 읽고/쓰는 함수 2개가 실제로 만들어진다 — 이 단계를 건너뛰면 앱은 접속할 데이터베이스가 아예 없는 상태가 된다.

  1. SQL Editor 접속 (URL 끝의 긴 코드는 저장된 쿼리 하나를 가리키는 것 — 다음번엔 그냥 왼쪽 메뉴의 SQL Editor를 새로 열면 됨)
  2. 위 스크립트 전체를 붙여넣기
  3. Run 버튼(또는 Cmd/Ctrl + Enter)으로 실행 — 한 번만 하면 됨, 다시 실행해도 if not exists 덕분에 안전함

확인하는 곳 → Table Editor에서 실제 저장된 가족 데이터 row 확인, SQL Editor에서 직접 쿼리 실행 가능

우리 앱은 이렇게 구성되어 있다

src/app/App.tsx ← 화면·로직 대부분이 여기 한 파일에 src/lib/supabaseClient.ts ← Supabase 접속 설정 src/lib/familySync.ts ← 가족 코드로 데이터 읽고/쓰는 함수 .env ← 로컬 개발용 비밀 키 (GitHub엔 안 올라감) public/ ← 아이콘·매니페스트·이 가이드처럼 그대로 서빙되는 파일

동작 흐름은 이렇다 — 브라우저에서 앱이 켜지면 가족 코드를 확인하고, Supabase에서 그 코드에 해당하는 데이터를 가져와 화면에 반영한다. 이후 뭔가 바뀌면(미션 완료, 설정 변경 등) 0.7초 뒤 자동으로 Supabase에 다시 저장된다.

09 · Operations

운영 FAQ — 실사용 중에 생기는 질문들

지금처럼 서비스가 이미 실사용 중인 상태에서, 새 기능을 추가하거나 데이터가 쌓일 때 실제로 어떻게 되는지 정리했다.

Q

메모 기능, 게임 요소 같은 새 기능을 요청하면 바로 앱에 반영되나요?

아니요, 자동은 아님

요청한다고 그 순간 실시간으로 바뀌지는 않는다. 매번 이 순서를 거친다:

  • 코드 작성
  • 로컬에서 실제 화면으로 동작 테스트
  • GitHub에 코드 반영 (커밋·푸시)
  • Vercel에 배포 (vercel --prod)

이 과정이 보통 몇 분 걸리고, 배포가 끝나야 실제 사이트(project-woohyun.vercel.app)에 새 기능이 보인다. 배포 전까지는 지금 쓰고 있는 화면이 그대로 유지되니, 운영 중에도 부담 없이 요청하면 된다.

Q

GitHub에 올리고 Vercel에 배포하는 작업을 제가 직접 해야 하나요?

아니요, 직접 안 해도 됨

기능을 요청하면 코드 작성부터 GitHub 반영, Vercel 배포까지 같은 대화 안에서 전부 처리된다. 직접 터미널을 열 필요는 없다.

다만 나중에 직접 해보고 싶거나, 다른 프로젝트에 이 방식을 다시 쓰고 싶다면 06번 "기술 구성" 섹션의 단계별 순서를 그대로 따라 하면 된다.

Q

그럼 GitHub에 커밋하는 건 그냥 관리·기록용인가요? (자동 배포가 안 되어 있다면서요)

네, 맞다 — 지금은 기록용

GitHub 커밋·푸시와 Vercel 배포는 현재 완전히 분리되어 있다. GitHub에 코드를 올리는 것만으로는 실제 사이트(project-woohyun.vercel.app)에 아무 변화가 없다 — 코드 변경 이력을 남기고 백업해두는 용도일 뿐이다.

그래서 기능 하나를 추가할 때마다 이 두 가지를 따로 실행한다:

  • GitHub에 커밋·푸시 — 기록 남기기
  • vercel --prod — 실제로 사이트에 반영하기
원래는 — GitHub에 푸시하면 Vercel이 자동으로 감지해서 바로 배포하는 게 정상적인 연결 방식이다. 하지만 처음 Vercel 프로젝트를 연결할 때 "GitHub 로그인 연결"이 실패해서, 지금은 이 두 단계를 수동으로 거치고 있다. 이 연결은 아직 안 되어 있고, 나중에 연결할 예정. 연결하고 나면 GitHub 푸시만으로 자동 배포되어 vercel --prod 단계가 필요 없어진다.
Q

기능을 추가하면 지금까지 쌓인 아이 데이터(레벨·사진·용돈)가 사라지거나 초기화되나요?

네, 안전합니다

코드(화면·기능)와 데이터(Supabase)는 완전히 분리되어 있다. 새 기능을 추가하거나 디자인을 바꾸는 건 "화면을 그리는 코드"만 바뀌는 것이고, 저장된 데이터에는 손대지 않는다. 배포가 끝나면 새 기능 + 기존 데이터가 같이 보인다.

Q

사진을 많이 찍으면 저장 용량이 꽉 차지 않나요? 미리 압축해서 올려야 하나요?

둘 다 이미 처리됨

압축은 자동으로 처리된다 — 교재 사진을 찍으면 앱이 자동으로 크기를 줄여서(가로 최대 640px, JPEG 압축) 저장한다. 따로 미리 압축하거나 신경 쓸 일은 없다.

용량 부족 경고도 추가했다 — 저장이 실패하면 조용히 사라지지 않고, 알림센터에 "저장 공간이 부족해요" 알림이 뜬다.

다만 오래된 사진을 자동으로 지우진 않는다 — 자동 삭제는 "학습 기록"이라는 기능 자체의 의미(한 학기 동안의 기록을 나중에 다시 보는 것)를 해칠 수 있어 일부러 넣지 않았다. 대신 학습 기록 화면에서 각 기록마다 "이 기록 삭제" 버튼으로 필요할 때 직접 골라 지울 수 있게 했다. 지워도 EXP·레벨·용돈에는 영향 없다.

Q

배포하는 도중에 아이가 앱을 쓰고 있으면 어떻게 되나요?

문제없다. 이미 열려있는 화면은 배포와 상관없이 그대로 동작한다. 새 버전은 그 화면을 다음에 새로고침하거나 다시 열 때부터 보인다 — 미션 도중에 갑자기 화면이 바뀌는 일은 없다.

Q

다른 컴퓨터에서도 이 프로젝트를 이어서 개선·수정하려면 어떻게 하나요?

준비물 확인

  • 그 컴퓨터에 VS Code 설치
  • VS Code에 Claude Code 확장 설치 후 로그인 (지금 쓰는 것과 같은 계정)
  • Git 설치 여부 확인 — 터미널에 git --version 입력해서 버전이 뜨면 이미 있는 것

1단계 — 코드 받기 (git clone)

  1. VS Code 실행 → 터미널 열기 (단축키 아래 표 참고)
  2. 코드를 받을 위치로 이동 (예: 바탕화면이면 cd Desktop)
  3. git clone https://github.com/jhkl0514/project-woohyun.git 입력 후 Enter
  4. 완료되면 project-woohyun 폴더가 그 자리에 생김
터미널 대신 마우스로 하고 싶다면 — 커맨드 팔레트(단축키 표 참고) 열고 Git: Clone 입력 → 위 GitHub 주소 붙여넣기 → 저장할 폴더 선택.

2단계 — 폴더 열기

  1. 파일 → 폴더 열기 (단축키 아래 표) → 방금 받은 project-woohyun 폴더 선택

3단계 — 패키지 설치

  1. 터미널 열고(Ctrl+`) pnpm install 입력
  2. "pnpm: command not found" 같은 오류가 뜨면 pnpm이 없는 것 — npm install -g pnpm으로 먼저 설치 후 다시 시도

4단계 — .env 파일 새로 만들기 (중요)

이 파일은 비밀 키가 들어있어서 일부러 GitHub에 올리지 않았다. 그래서 clone만 받으면 이 파일이 없는 상태 — 직접 만들어야 앱이 Supabase에 접속할 수 있다.

  1. VS Code 왼쪽 탐색기에서 project-woohyun 최상단 우클릭 → 새 파일(New File) → 이름을 정확히 .env로 입력
  2. Supabase API Keys 페이지에서 Project URL, anon key 복사 (06번 섹션 3단계와 동일)
  3. 아래 두 줄을 그대로 붙여넣고 값만 교체:
VITE_SUPABASE_URL=여기에_Project_URL_붙여넣기
VITE_SUPABASE_ANON_KEY=여기에_anon_key_붙여넣기

5단계 — 잘 되는지 확인

  1. 터미널에 pnpm run dev 입력
  2. "Local: http://localhost:5173" 같은 문구가 뜨면 성공 — 그 주소를 브라우저 주소창에 붙여넣어 화면이 뜨는지 확인
  3. 확인했으면 Ctrl+C로 서버 종료 (계속 개발할 땐 켜둬도 됨)
